# HTMX web view

Issue: #11 — "enhancement: use HTMX for the web view"

## Problem

The `-serve` dashboard works, but its front end is entirely hand-rolled and
lives inside Go source files.

`serve.go` is 34 KB, and roughly 250 lines of that are four `html/template`
sources stored as Go string constants (`pageTmpl`, `railTmpl`, `detailTmpl`,
`stepcardTmpl`). Editing markup means editing a Go raw string literal: no
syntax highlighting, no formatter, and a standing constraint that the content
can never contain a backtick — which is why `ui.go`'s header comment documents
that the client script is written as a classic script with no template
literals.

`ui.go` holds ~90 lines of bespoke client JavaScript: a keyed DOM `morph()`
reconciler, a 3-second `setInterval` that fetches `/rail` and `/detail`, a
byte-comparison short-circuit, transcript scroll pinning, disclosure-state
preservation, and header-stat patching through a hidden `#railmeta` element.
That reconciler exists for a real reason — replacing `innerHTML` on every poll
re-created every node, which replayed the `.fadein` entrance animation across
the whole screen, dropped text selection and focus, reset every `<details>`
disclosure the user had opened or closed, and reset transcript scroll offsets,
so one new transcript line looked like a full page reload. But it is
maintained, hand-written infrastructure that a mature library now provides.

Tailwind is already in use, via `<script src="https://cdn.tailwindcss.com">`
with an inline `tailwind.config` object. That is the browser-side JIT build,
which Tailwind documents as unsuitable for production and which leaves the
dashboard rendering unstyled with no network connection.

## Scope

In scope:

- Replace the hand-rolled poll/morph client with HTMX plus the official
  idiomorph morph extension, both vendored locally.
- Move templates and static assets out of Go string constants into a real
  `web/` folder, embedded with `go:embed` so the release stays a single binary.
- Replace the Tailwind CDN script with a vendored, prebuilt `app.css` committed
  to the repo, generated by the Tailwind standalone CLI.

Out of scope: SSE or websockets, dark mode, a dev-mode that loads templates
from disk without a rebuild, HTMX-driven ticket selection (plain full-page
links stay), vendoring web fonts, and any visual redesign. **The rendered
output should be pixel-identical to today's dashboard.**

## Decisions

Three questions were settled before design, and the alternatives are recorded
here because each was a live option.

**Tailwind delivery: vendor a prebuilt `app.css`.** The generated CSS is
committed to the repo and embedded in the binary. The alternative — a real
Tailwind build step wired into CI and `.goreleaser.yaml` — would make Node a
hard build dependency for a project that currently needs none, and would create
a stale-CSS failure mode for anyone running `go build` directly. Vendoring
keeps `go build` the only build command and matches the repo's Go-only CI
stance, at the cost of a manual regeneration step that must be documented and
guarded by a test.

**Morph strategy: HTMX plus the idiomorph extension.** HTMX's default
`hx-swap="innerHTML"` reintroduces every regression `morph()` was written to
prevent. `hx-preserve` does not solve it either: it needs a stable `id` on every
protected element and preserves them verbatim, so a preserved transcript would
stop receiving new lines. Idiomorph is a mature keyed morphing algorithm from
the same family as the hand-rolled one; it preserves node identity, so
disclosure state, scroll offsets and focus survive because the nodes are never
re-created. Roughly 90 lines of bespoke reconciler are deleted and about 40
lines of genuine policy are retained.

**Asset loading: embed-only.** Templates and assets live in `web/` as real
files, and `go:embed` puts them in the binary. Editing a template requires a
rebuild, but `go build` on this project takes about a second, which makes a
`-web-dir` dev flag — with its filesystem abstraction and second code path to
test — hard to justify.

## Architecture

### File layout

```
web/
  tailwind.css              # Tailwind v4 source: @import, @theme, @source, custom CSS
  templates/
    page.html              # {{define "page"}} — shell, header, rail/main containers
    rail.html              # {{define "rail"}}, {{define "statbar"}}, {{define "statbar-oob"}}
    detail.html            # {{define "detail"}}
    stepcard.html          # {{define "stepcard"}}
  static/
    htmx.min.js            # vendored htmx 2.x, pinned to an exact version
    idiomorph-ext.min.js   # vendored htmx morph extension (bundles idiomorph)
    app.css                # Tailwind CLI output — committed, regenerated manually
    app.js                 # residual client policy, replaces ui.go
```

### Go changes

**New `web.go`** owns the embedded filesystem and nothing else:

```go
//go:embed web/templates web/static
var webFS embed.FS
```

plus `fs.Sub` accessors for the templates and static subtrees, and the
`http.FileServerFS` handler for `/static/`.

**`serve.go` shrinks to HTTP concerns.** The four template constants are
deleted. The ~20 template helper functions and formatters (`money`, `dollars`,
`tokens`, `duration`, `short`, `shortid`, `ctxTokens`, `hasUsage`, `hasRunning`,
`errCount`, `statusChip`, `nodeClass`, `cardClass`, `divClass`, `stateKind`,
`stripeClass`, `txLine`, and the `FuncMap` that binds them) move to a new
`render.go`. `serve.go` is left with the `Server` type, `NewServer`, `Handler`,
the three handlers, `load`/`issues`/`backfillPR`/`summarize`, and `renderHTML` —
about 350 lines. This is a targeted cleanup of the file being rewritten, not a
general refactor.

**Template parsing simplifies.** `NewServer` currently parses three separate
template sets from concatenated constants, so `Server` carries three
`*template.Template` fields. With `ParseFS(webFS, "web/templates/*.html")` there
is one set containing all defines, and handlers select by name with
`ExecuteTemplate`. The three fields collapse to one.

**`ui.go` and `ui_client_test.go` are deleted.** Their replacement is
`web/static/app.js` and the Go-level tests below.

### Tailwind pipeline

`web/tailwind.css` is the source of truth, built with the **Tailwind v4
standalone CLI** — a single downloadable binary, so no npm, no `package.json`,
and no Node is introduced anywhere.

The inline `tailwind.config` object migrates into a CSS-first `@theme` block:
the `ink` / `panel` / `panel2` / `line` / `line2` / `text` / `muted` / `faint` /
`ok` / `err` / `warn` / `live` palette and the IBM Plex Sans / Mono font stacks.
The hand-written CSS below it — `@keyframes hb`, `ring`, `fadein`, the
`.node-ok` / `.node-err` / `.node-live` box shadows, the `details > summary`
marker suppression and `.chev` rotation, the `.scroll` scrollbar styling, and
the `prefers-reduced-motion` block — moves across verbatim.

**Class sources must include the Go files.** `stateKind`, `stripeClass`,
`nodeClass`, `cardClass`, `divClass` and `statusChip` construct Tailwind class
strings inside Go source. If the CLI scans only `web/templates/`, every class
that exists solely in Go is tree-shaken out and the dashboard renders
half-styled. So `tailwind.css` declares both:

```css
@source "./templates/*.html";
@source "../*.go";
```

Regeneration is manual. The command goes in the README and in a comment at the
top of `tailwind.css`:

```
tailwindcss -i web/tailwind.css -o web/static/app.css --minify
```

Google Fonts remains a CDN `<link>`, exactly as today. The existing
`system-ui` / `ui-monospace` fallbacks mean an offline dashboard degrades to
system fonts rather than breaking.

### HTMX wiring and data flow

Routes keep their current shape — `GET /`, `GET /rail`, `GET /detail` — with
`GET /static/` added, served from the embedded FS.

**Polling.** `<body hx-ext="morph">`. The rail and main panes carry
`hx-get="/rail?issue=N"` and `hx-get="/detail?issue=N"` with the selected issue
rendered server-side into the query (ticket selection is still a plain
full-page `<a href="/?issue=N">` link, so the URL is fixed for the life of the
page), `hx-trigger="every 3s"`, and `hx-swap="morph:innerHTML"`. The entire
`setInterval` / `fetch` / byte-compare block in `ui.go` disappears.

**Header stats.** The hidden `#railmeta` element and the `applyMeta` / `setText`
patching are removed. `handleRail` renders the rail fragment *and* a
`{{define "statbar-oob"}}` element carrying `hx-swap-oob="true"`, so htmx
relocates the header counters itself. The page header renders the same
`statbar` content inline on first paint. The statbar holds no user state, so a
plain outerHTML swap is correct there — it does not need morphing.

**`app.js` keeps only what morphing cannot express**, roughly 40 lines:

- the 1-second "live · Ns ago" ticker, reset on `htmx:afterRequest` rather than
  after a manual fetch;
- `copySid`, unchanged in behaviour;
- transcript bottom-pinning on `htmx:afterSwap`. Scroll offsets now survive for
  free, since idiomorph never re-creates the node, so this only re-pins feeds
  that were already at the bottom;
- the disclosure rule, below.

**The disclosure rule, and its one risk.** Today `ownedByClient()` refuses to
re-apply the server's `open` attribute to any element with `data-disc`, so a
user who closes a running step's transcript does not have it reopened three
seconds later. Under idiomorph this becomes a `beforeAttributeUpdated(name,
node)` callback returning `false` for `open` on `data-disc` nodes. Newly
appearing step cards still auto-open, because idiomorph inserts new nodes
wholesale rather than morphing them.

The risk is whether the htmx morph extension honours a callback installed
globally through `Idiomorph.defaults.callbacks`, or overrides it with its own
config object. **This must be verified during implementation, not assumed.** If
the extension overrides it, the fallback is to register a small custom htmx swap
that calls `Idiomorph.morph` directly with our config — identical behaviour,
about 10 extra lines.

## Error handling

Unchanged in character. `NewServer` still fails fast when a template does not
parse — now a `ParseFS` error rather than a `Parse` error. `renderHTML` still
renders into a buffer before touching the `ResponseWriter`, so a template error
yields a clean 500 with nothing already written, and a mid-poll client
disconnect (`EPIPE` / `ECONNRESET`) is still swallowed rather than logged as
noise. The GitHub TTL cache, the last-good-list fallback, the "GitHub
unreachable" banner and the PR backfill memoization are all untouched.

Embedding removes a failure mode rather than adding one: assets cannot be
missing at runtime.

## Testing

All tests are pure Go, so `go test ./...` remains the whole test story and the
`node` dependency disappears.

**Kept unchanged.** Every existing assertion in `serve_test.go`. These assert on
rendered HTML and survive the move to `ParseFS` untouched — they are the safety
net proving the rewrite did not change output. Any diff to these tests is a
signal that output drifted.

**New tests.**

- The page carries `hx-ext="morph"`, and the rail and main panes carry the
  expected `hx-get`, `hx-trigger="every 3s"` and `hx-swap="morph:innerHTML"`,
  with the selected issue number in the `hx-get` query.
- The `/rail` response contains the `hx-swap-oob` statbar with the correct
  ticket, running and spend values.
- Rail rows still carry `data-k="t<N>"` and step list items `data-k="s<seq>"`,
  so idiomorph can key them.
- `/static/htmx.min.js`, `/static/idiomorph-ext.min.js`, `/static/app.js` and
  `/static/app.css` each serve 200 with a non-empty body and a sane
  `Content-Type`; an unknown static path serves 404.
- **Stale-CSS guard:** `app.css` contains sentinel selectors that only exist if
  the CLI ran against both source globs — one class used only in a template and
  one produced only by a Go helper such as `nodeClass`. This is the guard
  against someone editing classes and forgetting the manual regeneration step.

**Deleted.** `ui_client_test.go` and its hand-rolled DOM shim, along with
`ui.go`. Their coverage was of a reconciler that no longer exists; the
equivalent guarantees are now either idiomorph's own (node identity) or covered
by the HTML assertions above.

**Manual verification checklist.** These are the behaviours the Go tests cannot
prove, and they must all be checked against a running `-serve` before the work
is called done:

1. Open a transcript disclosure; it stays open across at least three polls.
2. Close a *running* step's disclosure; it stays closed across at least three
   polls.
3. Scroll a transcript up; the offset holds while a separate bottom-pinned feed
   keeps following new lines.
4. No whole-screen fadein flash on poll — only genuinely new content animates.
5. With the network offline, the page still renders fully styled (system fonts
   substituting for IBM Plex is expected and acceptable).
6. Ticket selection, the GitHub-unreachable banner, the copy-session-id button
   and the two-column architect/answerer layout all behave as before.

## Documentation

The README's "Progress dashboard (`loope -serve`)" section gains a short note on
where the web assets live, that they are embedded so the binary stays
self-contained, and the exact Tailwind regeneration command with a warning that
it must be re-run whenever Tailwind classes change in either the templates or
the Go helpers.
